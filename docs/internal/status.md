### Deferred within consumer-side discovery

- **Template-literal / URL-builder paths** → opaque boundary binding, `lowConfidence` finding.
- **Multiple client calls per function** → first pass: one call site per consumer function, deduplicates on enclosing function.
- **Range-based status matching** → `status >= 400`, `response.ok` conditions produce opaque status predicates today; could be resolved to status ranges in a follow-up.
- **Recursive dependency extraction** → local function calls within the consumer are `invocation` effects, not recursively extracted into their own summaries.
- **axios, tRPC, GraphQL clients** → additive; the mechanism is proven, more packs are data.

### Deferred within Phase 8

- **`axios.create()` instances with `axios({ url, method })` bare-call form and `axios.request(config)`** — only per-verb method calls are matched.
- **Aliased default imports** (`import myAxios from "axios"`) — pack matches the conventional `import axios from "axios"` only.
- **Multi-hop wrapper expansion** (helper-of-helper) — single hop only.
- **Method-as-parameter wrappers** (`fetch(method, path)`) — only path-passthrough is recognised.
- **`BodyS3Location`** in CloudFormation — out-of-line OpenAPI bodies aren't fetched.
- **Raw CDK source** — consume `cdk synth` output instead.
- **Greedy `{proxy+}` paths** — extracted verbatim into the boundary path; the checker doesn't normalise them as wildcards, so consumer calls with concrete sub-paths won't pair against a `/{proxy+}` provider.
- **`!Sub` partial resolution** — `!Sub "${ResourceLogicalId.Arn}"` and parameter interpolations are passed through as their raw template strings. Doesn't matter for path/method extraction (paths are usually literal in practice) but does mean integration URIs lose their resource references.
- **`!ImportValue` cross-stack tracking** — silently passed through. Useful for graph extraction across stacks; not load-bearing for behavioural summaries within a single template.
- **Non-HTTP triggers** (`AWS::Lambda::EventSourceMapping`, `AWS::Events::Rule`, `AWS::SNS::Subscription`) — would each warrant their own resource-semantics package (`@suss/contract-aws-sqs`, `@suss/contract-aws-eventbridge`, etc.) following the same reader/semantics split as `aws-apigateway`. Out of scope while the focus is HTTP boundaries.

## Decisions log

1. **Single language per invocation.** The tool analyzes one tsconfig at a time. Monorepo orchestration is the user's responsibility.
2. **Framework packs are declarative data**, not code. They describe patterns; the adapter interprets them.
3. **The condition extractor is four independent functions.** Each testable with its own fixtures.
4. **Opaque predicates are explicit.** When decomposition fails, preserve source text and mark opaque. Never silently drop.
5. **Dependency resolution degrades in stages.** In-project → full extraction. Typed dependencies → type info. Untyped → opaque. No config.
6. **ts-morph is the TypeScript foundation.** Public, stable, AST + symbol table + type checker in one package. No SCIP, no LSP.
7. **`RawCodeStructure` is the adapter/extractor boundary.** Extractor never sees AST. Adapter never runs assembly logic.
8. **Maps over switch statements** for dispatch on discriminated unions. `Record<Kind, converter>` enforces exhaustiveness at the type level, no fall-through risk.
9. **Biome 2.x for linting and formatting.** No ESLint, no Prettier. Pre-commit hook (husky + lint-staged) runs `biome check --write` on staged files. `turbo lint` / `turbo lint:fix` available at the root. See `docs/style.md` for the full style guide.
10. **`expectedInput` flows through `RawBranch`, not post-assembly patching.** The adapter populates `RawBranch.expectedInput`; the extractor copies it to `Transition.expectedInput` during assembly. This preserves the pipeline contract (adapter → `RawCodeStructure` → extractor → `BehavioralSummary`) and means any future adapter (Python, etc.) gets field-level comparison for free.
11. **`"client"` vs `"consumer"` in `CodeUnitKind`.** `"consumer"` is for message consumers (Kafka, SQS). `"client"` is for API call sites (fetch, ts-rest `initClient`). Different behavioral models: clients branch on response status and read body fields; consumers receive messages and produce effects.
12. **Depth before breadth.** Cross-boundary analysis depth (field-level body comparison, predicate-level transition matching, automatic boundary pairing) takes priority over language breadth (Python adapter, React components). Breadth multiplies depth; adding depth later is a rewrite, adding breadth later is additive.
13. **Reduce opaqueness recursively.** When the extractor encounters something it can't decompose (call expression, unresolved reference), it should try to resolve through it — inline the function body, follow the variable chain, look up the type. Same recursive strategy at every level. Opaque predicates are the source-preserving fallback, not the first resort.
14. **`FrameworkPack` → `PatternPack` (done).** The interface describes discovery patterns, terminal extraction, and binding extraction. It applies equally to frameworks (Express, ts-rest) and runtime APIs (fetch). `@suss/runtime-web` uses the same interface. Renamed alongside response property semantics (7.9).
15. **Response property semantics belong in the pack, not the checker (done).** Properties like `.ok` (fetch) and `.body` (ts-rest) have framework/runtime-specific relationships to the HTTP status code and response body. The pack declares these semantics via `ResponsePropertyMapping[]` so the adapter resolves derived properties (`.ok` → `status >= 200 && status <= 299`) at extraction time. The checker never needs to know about framework-specific response shapes.
16. **Throw-with-status converts to response at extraction time.** When a framework pack extracts a status code from a thrown value, the extractor produces a `response` output, not a `throw` output. The behavioral contract (what the consumer sees) takes priority over the mechanism (how the code achieves it). This keeps transition IDs stable across throw↔direct-response refactors.
17. **Stubs produce the same summary shape as extracted code.** Summary generators that don't extract from source (`@suss/contract-openapi`, `@suss/contract-cloudformation`) emit the same `BehavioralSummary[]` shape as the extractor and carry `confidence.source: "stub"`. The cross-boundary checker pairs stub providers with extracted consumers without any source-aware logic — pairing is by `(method, normalisedPath)` regardless of where each side came from. This is what lets a TypeScript axios consumer be checked against a Stripe / GitHub / internal-team API whose handlers we can't extract from source.
18. **Pack-aware checker via summary metadata.** The body-compatibility checker can't be hardcoded to fetch's `.body` accessor — axios uses `.data`, ts-rest uses `.body`, custom packs vary. Adapter writes `metadata.http.bodyAccessors` on each client summary from the pack's response semantics; checker reads it. Same pattern scales to other pack-specific knowledge (status accessor names via `metadata.http.statusAccessors`, request body accessors, etc.). Keys live under `metadata.http.*` — see decision #25.
19. **Wrapper expansion is a post-pass over the discovered summaries.** Path-passthrough wrappers (`getJson(path)` forwarding to `axios.get(path)`) are recognised after the per-file discovery completes. ts-morph references find callers across the project; each caller gets a synthetic `DiscoveredUnit` fed back through `extractCodeStructure` with a synthetic pack. This reuses every existing analysis (terminal extraction, branch tracking, expectedInput) instead of growing a parallel synthesis path.
20. **Stub doctrine is documented separately.** Reader/semantics layering (manifest reader builds normalized config → resource-semantics package emits summaries), the opaque-predicate naming convention for transcribed external contracts (`<vendor>:<service>:<contract>`), per-transition `metadata` for provenance, and the rule that multiple platform causes for the same status collapse into one transition rather than forcing consumer disambiguation — see [`docs/stubs.md`](stubs.md). This extends decision #17 with concrete conventions so additional stubs (CDK synth, Terraform, vendor-doc transcriptions) follow the same shape.
21. **`Transition.metadata` is a top-level field.** Optional `Record<string, unknown>` on every `Transition`, mirroring `BehavioralSummary.metadata`. Used by stubs to attribute platform-injected transitions to their source manifest, by the checker as ignored opaque payload, and by inspect/diff to render provenance. Added in IR alongside the stub doctrine work — additive, doesn't affect checker semantics.
22. **`BOUNDARY_ROLE` lookup replaces hardcoded provider/consumer kind sets.** The pairing logic previously enumerated `PROVIDER_KINDS = {handler, loader, action, middleware, resolver}` and silently dropped `worker`, `component`, `hook` into `noBinding`. Replaced with a single `Record<CodeUnitKind, BoundaryRole>` exported from `@suss/behavioral-ir` so adding a new code-unit kind produces a TS error if its role is missing, and so the classification has one source of truth across packages.
23. **zod is the IR's single source of truth.** Behavioral summary types are derived from zod schemas in `packages/ir/src/schemas.ts` via `z.infer`, the JSON Schema published for non-TS consumers is generated from the same schemas at build time (`z.toJSONSchema`), and the runtime parsers (`parseSummaries`, `safeParseSummaries`) are exposed for boundary validation in CLI and downstream tools. This replaces the prior hand-written-types + hand-built-JSON-Schema split (kept aligned only by an Ajv test). The "zero dependencies" line on `@suss/behavioral-ir` is gone — it now has one peer dep on `zod ^4.0.0`. The trade-off: bundlers that inline transitively-reached generic types can produce large `.d.ts` files; `safeParse*` returns are explicitly annotated to keep declarations small (~3 KB instead of ~1 MB).
24. **Status property names come from the pack, not from a hardcoded list.** The checker previously matched only `"status"` and `"statusCode"` when finding status-eq predicates in consumer transitions. Now the adapter writes `metadata.http.statusAccessors` on each client summary from the pack's `responseSemantics` (entries with `semantics.type: "statusCode"`), parallel to `metadata.http.bodyAccessors`. The checker reads `statusAccessorsFor(summary)` once per check and passes a `ReadonlySet<string>` through `consumerExpectedStatuses`/`refLooksLikeStatus`. Falls back to the historical `["status", "statusCode"]` for hand-written summaries or summaries produced before this metadata existed. Also dedupes the duplicated `isStatusRef`/`isStatusName` helpers that lived in both `response-match.ts` and `provider-coverage.ts`.
25. **Summary metadata is namespaced by boundary semantics; layered-boundary design is the north star.** HTTP-scoped metadata keys (`declaredContract`, `bodyAccessors`, `statusAccessors`) live under `metadata.http.*` rather than at the top level of `metadata`. This signals scope so future GraphQL / Lambda-invoke / queue-message packs can own sibling namespaces (`metadata.graphql.*`, `metadata.lambda.*`, etc.) without migration. Semantics-neutral keys (e.g. `metadata.derivedFromWrapper` wrapper-expansion provenance) stay at the top level. The checker is still HTTP-coupled end-to-end — status codes as discriminator, response body as payload, `(method, normalizedPath)` as pairing key, 2xx success range. Refactoring that coupling into a `BoundarySemantics` interface waits for a second concrete semantics to forcing-function the seams (GraphQL is the planned first target because it shares transport with REST but has different discriminator / payload / pairing shape). See [`docs/boundary-semantics.md`](boundary-semantics.md) for the layered transport / semantics / recognition model and what the refactor will look like when it lands.
26. **`@suss/contract-openapi` accepts OpenAPI 3.1.** Spec-compatibility upgrades: `type` can be a string or an array (JSON Schema 2020-12 style), with `"null"` in the array meaning nullable — 3.0's `nullable: true` and 3.1's `type: [..., "null"]` normalize to the same `union<T, null>`. Adds `const` as a single-value enum shorthand. Adds `discriminator` support on `oneOf`/`anyOf`: when a mapping is declared, each variant's discriminator property is narrowed to the literal that maps to it, so the cross-boundary checker can use discriminator-value as a distinguishing field. Range status codes (`1XX`-`5XX`, case-insensitive) are expanded into transitions with `statusCode: null` and `metadata.http.statusRange: { min, max, spec }` — the IR has no range variant on `ValueRef`, so the range lives in per-transition metadata for inspect and future range-aware checker passes to consume.
27. **CI auto-refreshes tracked coverage.** The committed `coverage-summary.json` / badge SVGs are maintained by CI instead of locally: on every push (main or same-repo PR branch), CI regenerates them and pushes the refresh back with a skip-CI marker. PR builds check for *coverage regression* against main's baseline rather than file-diff drift — drifting files are expected (the auto-commit handles them); a line-coverage percentage below main for any package fails the build. Fork PRs lack write permission and silently no-op the refresh; fork contributors regenerate locally. Pre-push hook no longer runs `npm run badges` — its output never made it into the pushed commits anyway.
28. **Summary-level confidence is informational, not prescriptive.** `ConfidenceInfo.level` on `BehavioralSummary` measures the extractor's analysis quality on one side of a boundary (opaque-to-structured predicate ratio, wrapper-expansion downgrade, etc.). The checker does *not* downgrade finding severities based on it — the `lowConfidence` finding kind is the correct mechanism for per-finding uncertainty, and conflating the two axes would hide both. The human `suss check` output surfaces `(confidence: medium|low)` next to a provider or consumer side whose confidence is below `high`, so reviewers can weigh findings themselves; downstream tools can read `summary.confidence` from the JSON output and apply their own policy. See the note in [`cross-boundary-checking.md`](cross-boundary-checking.md) — "Confidence is informational, not prescriptive."
29. **Overlapping-provider findings are deduped; divergence is a separate task.** When two or more providers describe the same boundary (OpenAPI stub + CloudFormation stub + extracted handler for the same REST endpoint), each pairs independently with every consumer — producing identical findings per source. `checkAll` now runs a dedup post-pass that collapses findings agreeing on `(kind, boundaryKey, description, consumer.summary, consumer.transitionId)` into a single representative carrying the full list of contributing provider-summary identifiers in `Finding.sources`. `checkPair` is unchanged (one pair at a time still emits raw findings; users composing their own pipelines run `dedupeFindings` themselves). Divergence — when sources *disagree* about whether a status exists, what body shape it has, etc. — is out of scope here and tracked as a separate follow-up; the planned shape anchors the comparison on the declared contract (each provider is compared against `metadata.http.declaredContract`, not pairwise across providers) so it reuses existing infrastructure and attributes each disagreement to one source.
30. **`.sussignore` for accepted findings.** A standalone `.sussignore.yml` (or `.sussignore.yaml` / `.sussignore.json`) at the project root lists rules that match findings by `(kind, boundary, consumer.summary, consumer.transitionId)` and apply one of three effects: `mark` (default — shown with `suppressed (mark): <reason>` annotation, excluded from exit-code threshold), `downgrade` (severity drops one level, still counts toward threshold at the downgraded level), `hide` (filtered from output entirely). `reason` is a required field. Deliberately **no** `expires` feature — cargo-cult dates and soft warnings don't solve suppression rot; the substantive mitigation is the required `reason` + future observability (a `suss suppressions list` subcommand + CI-visible suppression counts). Narrow-scope matching (requires kind AND boundary/transitionId) is the default; broad-scope (`scope: "broad"`) opts in to kind-only or boundary-only matches. Alternative designs considered: in-source pragmas (rejected — findings emerge from cross-boundary comparison, no single line to pin them to); per-summary metadata (rejected — summaries are regenerated, unclear source of truth). `@suss/checker` owns the rule types + matching logic; `@suss/cli` handles the file I/O.
31. **Cross-source contract agreement is a two-layer model.** Layer 1 (`checkContractConsistency`) compares each provider's transitions against *its own* declared contract — "is this provider doing what it says it does?" Layer 2 (`checkContractAgreement`) compares the declared contracts *across sources at the same boundary* — "do the OpenAPI spec and the CFN template agree on what statuses exist?" Layer 2 operates on `{statusCode, body}` tuples only, not transitions, so the comparison is strictly simpler than provider-vs-provider transition comparison would be (no platform-injected transitions, no conditions, no gap logic). Emits a new `contractDisagreement` finding kind; one finding per non-unanimous status at a boundary, not N×(N-1)/2 pairwise. Layer 1 needed a fix for this to work without false positives: the `declaredContract.provenance` marker (`"derived"` vs `"independent"`) lets a pack say whether its contract and transitions come from the same source — OpenAPI stubs are `"derived"` (self-comparison is tautological and skipped); ts-rest handlers and CFN stubs are `"independent"`. Stubs now populate `declaredContract`, which they previously didn't — a prerequisite for Layer 2. Supersedes earlier "sourceDiscrepancy finding kind" planning — contract-to-contract is strictly simpler than transition-to-transition and handles 3+ sources identically to 2.
32. **Strategic direction: React first, multi-shape contracts as the concepts vision.** Suss has shipped HTTP as its only concrete boundary and has treated "contract" as synonymous with "schema" (OpenAPI, ts-rest `responses`, CFN `MethodResponses`). Both assumptions need breaking to honor the platform positioning. React is the next boundary we'll ship — forcing function for the `BoundarySemantics` refactor, highest adoption reach, hardest declaration density (no `defineEmits` / `defineSlots` the way Vue has). In React-land, no single contract shape is sufficient: Storybook stories capture canonical scenarios, snapshots capture output observations, Playwright captures behavioral observations, Figma captures design intent. The inferred summary crossed against all four is the full contract-consistency picture. New docs: [`docs/contracts.md`](contracts.md) (the five-shape taxonomy — schema / examples / tests / snapshots / design — with epistemic characters: specification, observation, derivation) and [`docs/roadmap-react.md`](roadmap-react.md) (phased plan: Phase 1 inferred extractor, Phase 2 Storybook stub, Phase 3 Figma stub, Phase 4 cross-shape contract agreement; what the rollout commits us to and what it defers). Design-only for now — no code yet.
33. **React Phase 1 in progress.** Phase 1.1 shipped: `@suss/framework-react` recognizes default-exported function components and classifies JSX-element / fragment / self-closing / parenthesized-JSX returns as `render` terminals carrying the root element name. `jsxReturn` TerminalMatch variant added to `@suss/extractor`; adapter's `tryMatchJsxReturn` extracts the root name into `RawTerminal.component`. Side fix: default-export discovery now preserves the function's own name (`export default function UserCard() {}` → identity "UserCard") instead of collapsing everything to "default" across files. Phase 1.2 shipped: `componentProps` input-mapping variant — walks the first parameter's `ObjectBindingPattern` and emits one `Input` per destructured name (role = name, typeText from the type checker), falling back to a single whole-object Input (role "props") when the parameter isn't destructured. Existing hook-recognition via `extractDependencyCalls` captures `useState`/`useEffect`/etc. for free because they're variable-declaration-with-call patterns — they surface in the summary only when downstream constructs (conditions, output expressions) reference the hook's return value. Deferred to later phases: nested render-tree capture, event handler body analysis, conditional-rendering (`cond && <X/>`), custom hooks, class components.

34. **React Phase 1.6 shipped — nested render tree.** `Output.render` grew a `root?: RenderNode` field where `RenderNode` is a recursive discriminated union `{ type: "element"; tag; attrs?; children } | { type: "text"; value } | { type: "expression"; sourceText }`. Adapter's `jsxToRenderNode` walks JSX elements / fragments / self-closing tags and their children, preserving source order; whitespace-only text is stripped; dynamic JSX expression children carry their original source text so downstream tools can display them without re-parsing. Phase 1.6 is a pure IR extension — checker behavior unchanged; sets up Phase 1.6b (handler references on element nodes) and Phase 2 (Storybook story trees can be compared node-for-node against inferred render trees).

35. **React components are N code units, not one lifecycle blob.** A component file yields one `component`-kind summary (render body) plus, in later phases, one `handler`-kind summary per event-handler prop and one `effect`-kind summary per `useEffect`. They share the component's identity prefix (module + export name) and share state via closed-over hooks, but have distinct inputs (props vs event vs deps), distinct outputs (JSX vs setState/callback invocations vs side-effects), and fire at distinct times. This is consistent with how HTTP already handles multi-registration per file — one Express handler module registers N `app.get/post/...` code units sharing a source file. The `BoundaryBinding.semantics` layer distinguishes them (`render` vs `handler(name)` vs `effect(index)`). Rejected alternative: one big "component lifecycle summary" conflating everything — conflates semantically different events and loses the information that makes React findings useful.

36. **Jackson concept-design framework is the theoretical ground for the multi-code-unit decision.** Daniel Jackson's concept design (MIT; "Concept Design Moves" NFM 2022, "What You See Is What It Does" SPLASH 2025) describes software as *concepts* (self-contained units with state + actions + purpose) linked by *synchronizations* (rules of the form "when action A₁ in concept C₁ fires, action A₂ in concept C₂ fires"). Syncs can only restrict behavior, never enable new actions. Structural mapping: a suss code unit ≈ a concept's action, shared closure/hook state ≈ the concept's state, boundary binding + shared-state reference ≈ a synchronization, a React component-cluster ≈ a concept instance. Deliberate divergences: (1) intent/purpose is absent — suss infers behavior bottom-up; `contractDisagreement` findings are the closest proxy for "purpose violated"; (2) genericity is absent — suss summaries are instance-specific, not reusable design primitives. We don't adopt the vocabulary ("concept") wholesale since overclaiming would imply intent declarations we don't support, but the structural alignment justifies treating React components as N units (and, by induction, any future code-cluster framework — background workers sharing a queue, etc.). Referenced in `docs/roadmap-react.md` and `docs/contracts.md`.

37. **React Phase 1.5 shipped — event handlers as separate code units.** Each JSX event-handler prop (`onClick`, `onChange`, etc.) that resolves to a locally-authored function becomes its own `handler`-kind `BehavioralSummary` sharing the component's identity prefix. Names: `ComponentName.functionName` for named locals; `ComponentName.tag.propName` for inline anonymous arrows, with `#N` suffixes when multiple anonymous handlers share the same `(tag, propName)` key. Prop-delegating references (`onClick={props.onDelete}`) are explicitly skipped — they're invocations of external handlers we don't own. Implementation is a post-pass in the TS adapter (`synthesizeReactHandlers` in `packages/adapter/typescript/src/adapter.ts`, plus `findHandlerLocations` in `react-handlers.ts`) triggered when a `component`-kind summary has `framework: "react"` boundary binding; the pattern-pack interface stays declarative. Discovery walks the component's JSX descendants without recursing into nested function bodies, and resolves identifier references to their local declaration sites to distinguish locally-authored handlers from prop delegations. `metadata.react` records parent component, element tag, prop name, and (for named locals) the declaration's own name. Handler summaries carry a placeholder `{ protocol: "react", framework: "react" }` boundary binding until the `BoundarySemantics` refactor lands. Effects on handler bodies are still limited to what variable-declaration-based dependency extraction catches — bare call statements (`setCount(n); onChange(n);`) aren't yet recorded as `RawEffect`s; that's a Phase 1.5b follow-up.

38. **React Phase 1.4 shipped — inline JSX conditionals as structured render-tree nodes.** `RenderNode` gained a `conditional` variant (`{ type: "conditional"; condition: string; then: RenderNode; else: RenderNode | null }`). Adapter's `jsxExpressionToRenderNode` decomposes `{cond && <X/>}` → `conditional(cond, <X/>, null)`, `{cond ? <A/> : <B/>}` → `conditional(cond, <A/>, <B/>)`, `{cond ? <X/> : null}` → `conditional(cond, <X/>, null)`, and recognises `false` / `undefined` sentinels (React's own no-render conventions) the same way as `null`. `{cond ? nonJsx : <Fallback/>}` promotes the JSX branch with the condition wrapped in `!(...)` textually. `||`, `.map()` calls, and identifier references stay as opaque `expression` nodes — decomposition only fires when at least one branch is statically JSX, so "we parsed this into structure" never overstates what the adapter actually understood. Condition text is preserved verbatim for v0; running the existing `parseConditionExpression` against it to produce a structured `Predicate` is a follow-up that unblocks cross-shape checking. Pure IR + adapter change; checker behavior unchanged.

39. **`PatternPack.subUnits` is the extension point for framework runtime scheduling.** A pack declares an optional `subUnits(parent, ctx) => DiscoveredSubUnit[]` hook; the TS adapter calls it for each discovered unit whose pack defines it, piping returned units through `extractCodeStructure + assembleSummary` with their own terminals / input mapping. Sub-units inherit the parent's boundary binding. Used by the React pack to synthesise event-handler and `useEffect` sub-units without the adapter hardcoding React-specific strings (`onClick`, `useEffect`, `"react"`). Generalises to: Node `EventEmitter` `.on()` handlers, React class methods, Vue lifecycle hooks, GraphQL field resolvers — anywhere a runtime schedules user callbacks from one source-level construct. Context interface `TsSubUnitContext` is exposed by `@suss/adapter-typescript` with primitives for JSX-attribute walking, call-expression-by-name walking, identifier resolution, array-literal reading. The pack-vs-language split is now enforced structurally: the adapter contains no framework-specific strings; React semantics live entirely in `@suss/framework-react`.

40. **`PatternPack.protocol` is required and names the transport class, not the framework.** Each pack explicitly declares what transport its boundaries cross — `"http"` for HTTP-transported (REST / OpenAPI / ts-rest / Express / Fastify / React Router / fetch / axios), `"in-process"` for boundaries without a network hop (React components and their sub-units, eventually custom hooks / module-internal cross-unit work), with a path to `"aws-sdk"` / `"queue"` / `"grpc"` when future packs need them. `BoundaryBinding.framework` carries the framework identity separately (`"react"`, `"express"`, etc.). The rename matters because `protocol: "react"` would have been a category error — React has no wire protocol; it has an in-process runtime. Making the field required and reserving it for transport keeps the IR's protocol-vs-framework distinction explicit even before the `BoundarySemantics` refactor fully splits transport from semantics. Guidance for pack authors lives in [`docs/framework-packs.md`](framework-packs.md#protocol).

41. **Refactor: React sub-unit synthesis moved from `@suss/adapter-typescript` to `@suss/framework-react`.** Supersedes the Phase 1.5 placement described in decision #37. The old `react-handlers.ts` file and the `synthesizeReactHandlers` adapter post-pass (gated on `framework === "react"`) are gone; handler + `useEffect` discovery now lives in `@suss/framework-react/src/sub-units.ts`, registered via `reactFramework().subUnits`. The adapter's replacement post-pass (`synthesizeSubUnits`) is framework-agnostic — it dispatches to whichever pack's `subUnits` hook is defined. `@suss/framework-react` now depends on `@suss/adapter-typescript` (for the context type + primitives) — an explicit "this pack is for React via TypeScript" coupling that other frameworks don't take on unless they need AST-level sub-unit work.

42. **React Phase 1.5b shipped — effect-body capture + fall-through terminals.** Two pieces working together: (a) a new `TerminalMatch` variant `{ type: "functionFallthrough" }` that packs can include to synthesise a default transition when no explicit return/throw covers the function's exit path; (b) a new adapter pass `extractInvocationEffects` that walks for bare `ExpressionStatement > CallExpression` and records each as a `RawEffect` `{ type: "invocation"; callee; async }`, attached to the default branch. Fall-through is pack-opt-in rather than always-on because the same language fact — every JS function implicitly returns `undefined` — means different things per pack: an HTTP handler that doesn't respond is a *bug* (keep `transitions: []` so the gap detector flags it), a React event handler with no explicit return is *normal* (synthesise a default transition carrying the body's side effects). Sub-unit scaffolding in the adapter's `DEFAULT_SUB_UNIT_TERMINALS` includes the fall-through match so every React handler / `useEffect` body gets a default transition without the pack re-declaring it. Calls whose source line coincides with a matched terminal (e.g. Express's `res.json(body)` in expression-statement position) are filtered out so a single source line never double-counts as both terminal and effect. V0 attribution: all invocation effects go on the default branch; branch-scoped attribution (effects inside an `if`/`for` block) is a Phase 1.5c follow-up.

43. **React Phase 1.6b shipped — JSX attributes on render-tree element nodes.** `RenderNode.element` gained optional `attrs?: Record<string, string>`, mapping each JSX attribute name to its raw value source text. String-literal attributes keep their quotes (`type="button"` → `"\"button\""`); expression-valued attributes record the full expression text (`onClick={() => setCount(c+1)}` → the arrow source); boolean-shorthand attributes (`<input disabled>`) map to `""`; spread attributes (`{...props}`) surface as `...exprText` keys. No interpretation happens in the adapter — the render tree just captures attribute shape. Consumers that care about React event semantics (the eventual Storybook-stub matcher, the cross-shape checker) combine `attrs` with the React pack's handler naming rule to resolve summary references on demand. Keeps adapter / pack responsibilities clean: adapter records what JSX says, pack interprets what React means. `attrs` is omitted entirely when an element has no attributes, keeping attribute-free markup terse.

46. **Phase A shipped — three-layer `BoundaryBinding`.** `BoundaryBinding` now carries `transport` (wire: `"http"`, `"in-process"`, …), `semantics` (discriminated union: `{ name: "rest"; method; path; declaredResponses? }` or `{ name: "function-call"; module?; exportName? }`), and `recognition` (pack identity: `"axios"`, `"ts-rest"`, `"react"`, `"openapi"`, …). The flat `{ protocol, method?, path?, framework, declaredResponses? }` shape is gone. `@suss/behavioral-ir` exports `restBinding` and `functionCallBinding` builders; every pack and stub uses them instead of object literals. Checker pairing dispatches on `semantics.name` — REST semantics pair by `(method, normalizedPath)`, function-call semantics don't pair automatically (escape-hatch for in-process units like React components and Storybook stubs). Wrapper-expansion flags unresolved forwarders via empty `method` / `path` on `rest` semantics rather than optional fields. `BoundarySemantics<S>` dispatch interface (see `docs/boundary-semantics.md`) is designed but not yet extracted — inline narrows on `semantics.name === "rest"` are adequate for one dominant variant; the abstraction lands when GraphQL does. 50. **Composed ts-rest contracts resolve through `PropertyAccessExpression`.** `resolveContractObject` in `packages/adapter/typescript/src/contract.ts` previously only handled `Identifier` and direct `ObjectLiteralExpression` for the `s.router(<contract>, {...})` first argument. Real-world ts-rest servers compose a top-level contract from sub-namespaces (`c.router({ internal: internalApi, events: eventsApi })`) and register handlers against a property access (`s.router(apiContract.internal, {...})`). Those handlers silently degraded to `function-call` semantics with `recognition: "ts-rest"` because the resolver bailed — the pack matched but contract reading returned null. Fix recurses the resolver into the base identifier, picks the named property from its routes literal, and follows through when the value is another identifier, inline object, or `c.router()` call. Fixture at `fixtures/ts-rest-composed/` mirrors the shape. Surfaced by running the adapter against an external production ts-rest service that had 13 handlers all falling through to `function-call`.

51. **`EffectArg` is a five-variant union that preserves shape even under opacity.** Originally limited to `string`/`number`/`boolean`/`object`/`array`/`template`/`null`. Extended with `identifier` (`{ kind: "identifier"; name }` — captures bare variables, property-access chains `user.profile.email`, element-access `config["host"]`, and `process.env.X` references as the full source text) and `call` (`{ kind: "call"; callee; args }` — recursive with bounded depth for `log(formatError(e))` composition). Object/array branches now preserve their *shape* when every value is opaque instead of collapsing to `null` — the keys a caller supplied are information about intent even when the values aren't resolvable. Shorthand-property assignments (`{ userId, count }`) expand to the identifier form at each key. Measured on a real external codebase: top-level null-arg ratio dropped from 45.6% → 1.5% across the full recursive arg tree. IR schema is `args: z.array(z.unknown())` so the extension is a TypeScript-type-only change at the IR layer; no migration needed.

52. **Transitive closure from entry points is the default library-discovery mechanism.** A user-facing question — "why do we need a `DiscoveryMatch` variant for internal orchestrators, isn't function-call already a boundary?" — forced a rethink of the original pack-configured approach. Replaced with a post-pass in `extractAll()` (`expandReachableClosure` in `packages/adapter/typescript/src/reachable-closure.ts`): every function statically reachable through a `CallExpression` chain from a pack-discovered unit becomes its own `library` summary with `boundaryBinding: function-call { transport: "in-process", recognition: "reachable" }`. The closure is a BFS worklist — newly-reached functions feed back into the frontier, so handler → A → B → C all get summaries. Same-project only (skips node_modules + declaration files + ambient `declare function` decls). Dedup by function node position avoids double-coverage against pack-produced summaries. Gated by `TypeScriptAdapterConfig.includeReachable` (default `true`). New `allPositional` InputMappingPattern variant emits one Input per declared parameter (role = name) for reachable library units — there's no framework-declared role space to draw from. Result on the nimbus integration-handler: 14 → 30 summaries, with the 16 additions being exactly the orchestrators a prior AI-quiz pass had flagged as invisible.

53. **Throw-terminal message extraction.** `tryMatchThrowExpression` in `packages/adapter/typescript/src/terminals.ts` hard-coded `message: null`. New helper `extractThrowMessage` pulls the first string-literal argument from the thrown constructor's args regardless of position — covers both the dominant `throw new Error("msg")` shape and the `throw new HttpError(404, "msg")` form where the message is the second arg. Template literals with substitutions preserve their source text (backticks + `${}`) so `` `Service ${name} not found` `` stays visible. Returns null for bare-identifier re-throws (`throw err` from a catch block) where no static message is resolvable — tracked as a follow-up (re-throw resolution would require walking the enclosing try block's call sites and consulting their summaries' throw terminals, cross-summary enrichment). Nimbus throw-message coverage: 0/7 → 4/7, with the remaining 3 nulls legitimately unresolvable statically.
